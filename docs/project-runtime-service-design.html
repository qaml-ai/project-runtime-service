Design direction

Extract the sandbox runtime into a boring, fast project compute service.

The main app should own product state and agent orchestration. The runtime service should own containers, project disks, command execution, file operations, copy-on-write clones, backups, quotas, disk growth, and project-scoped outbound capabilities.

Recommended path

Keep the new explicit Projects model, but add a sandbox-host backend that talks to our Go service. Then extract that Go service into its own repository once the API stabilizes.

XFS reflink clones gVisor containers project quotas generic egress proxy lazy container start

Architecture

The clean split is a narrow runtime API between camelAI and one or more compute backends. Cloudflare Sandbox, exe.dev, and our Go sandbox-host should all implement the same conceptual contract.

Agent Calls js_exec, file tools, bash, project clone, and project list.
Main Worker Checks auth, stores project metadata, picks backend, signs runtime requests.
Backend Adapter Maps project operations to Cloudflare, exe.dev, or sandbox-host.
Runtime Service Owns containers, filesystems, quotas, proxies, clone, backup, stop/start.
Project Disk Persistent working tree mounted at one fixed path inside the container.

Responsibilities

Production already has the key primitives: local XFS, project quotas, gVisor containers, and reflink=1. That means project clone can be a metadata operation instead of a backup/restore workflow.

Generic authenticated egress proxy

The runtime service should expose named outbound capabilities to a project. A sandbox can use a capability, but it cannot read the credential or create new capabilities. 

{
  "id": "artifacts-main",
  "projectId": "project_123",
  "targetBaseUrl": "https://artifacts.cloudflare.net",
  "allowedHosts": ["artifacts.cloudflare.net", "*.artifacts.cloudflare.net"],
  "allowedMethods": ["GET", "POST", "PUT", "PATCH", "DELETE"],
  "auth": {
    "type": "bearer",
    "credentialRef": "workspace.cloudflare_artifacts_token"
  },
  "headerPolicy": {
    "stripIncoming": ["authorization", "cookie", "x-forwarded-for"],
    "set": {
      "authorization": "Bearer ${credential}",
      "user-agent": "camelAI-runtime-proxy"
    }
  }
}
The generic part is request validation, path joining, host allowlisting, header policy, credential injection, streaming, and audit logging. Provider-specific logic should live in small auth plugins or request signers.

Proxy auth plugins

  • none: allowlist-only outbound access.
  • bearer: inject Authorization: Bearer ....
  • basic: inject HTTP Basic auth for Git remotes.
  • static_header: inject one or more configured headers.
  • aws_sigv4: sign S3/R2-compatible API requests.
  • oauth_bearer: resolve a refreshed token from a credential reference.
The sandbox must never be able to change the target URL, allowed hosts, or credential reference.

exe.dev integration pattern

exe.dev calls these network-mounted capabilities “integrations.” The core abstraction is not a Cloudflare-specific proxy. It is a named internal hostname attached to one or more VMs. The VM talks to http://<integration-name>.int.exe.xyz/, and exe.dev forwards the request to a configured target while injecting authentication outside the VM.

exe.dev concept What it does Equivalent in our runtime service
integrations add http-proxy Creates a named HTTP(S) proxy with injected headers, bearer token, basic auth, no-auth, or peer auth. POST /v1/projects/:id/proxies with auth plugin and header policy.
--attach vm:name Makes an integration available only to a specific VM. Project-scoped capability attached to one project/container identity.
--attach tag:prod Makes an integration available to every VM with a tag. Capability policy attached to project labels or workspace groups.
--attach auto:all Makes an integration available to all current and future personal VMs. Org/workspace default capability template.
<name>.int.exe.xyz Stable internal DNS name used from inside the VM. http://<capability>.proxy.local/ or http://runtime.internal/p/:id/.
--peer VM-to-VM proxy with generated auth and platform-set caller header. Project-to-project capability with generated service token and host-set caller identity.
GitHub integration Private Git access through an internal integration hostname, without storing GitHub tokens in the VM. Generic Git HTTPS proxy backed by credential refs or provider app tokens.
The design lesson: expose capabilities through stable internal names, attach them by policy, and inject credentials in the runtime service. The sandbox receives network access, not secrets.

Storage model

Each project gets one host directory on the sandbox disk. The directory is mounted into the container at a fixed path, ideally /workspace/project for the new Projects model.

  • Clone: stop or briefly pause source, run sync, reflink-copy to temp dir, apply quota, atomic rename.
  • Start: create container lazily on first command or file operation.
  • Stop: stop idle containers aggressively while leaving project disk intact.
  • Backup: async stream to object storage using restic, kopia, or custom tar/zstd.
  • Growth: monitor free space and grow Azure data disk before the host is pressured.

Clone flow

POST /v1/projects/project_b/clone

source = project_a
target = project_b

1. lock source and target
2. reject if target already exists
3. stop or pause source container briefly
4. sync source filesystem
5. cp -a --reflink=always source target.tmp
6. configure target project quota
7. mv target.tmp target
8. release locks
9. return timings

Do not silently fall back to a slow full copy for the fast clone API. If reflink is unavailable, return a clear capability error.

Runtime service API shape

Endpoint Purpose Notes
GET /v1/host/capabilities Report clone mode, filesystem, quota support, proxy auth plugins, max disk. Main app uses this to decide whether fast clone is supported.
POST /v1/projects/:id/ensure Create project disk and optionally start container. Should be idempotent.
POST /v1/projects/:id/exec Run a command in the project container. Starts container lazily if needed.
GET/PUT /v1/projects/:id/files/* Read, write, list, delete, move files. Use streaming for large files.
POST /v1/projects/:id/clone Create a copy-on-write clone from another project. Destination container does not need to start.
POST /v1/projects/:id/proxies Provision project-scoped outbound capability. Only main app can call this.
ANY /p/:capabilityId/* Runtime egress proxy used from inside the sandbox. Verifies caller project before forwarding.
POST /v1/projects/:id/stop Stop a running project container. Disk remains intact.

Security invariants

  • Sandbox code cannot read provider credentials.
  • Sandbox code cannot create or edit proxy capabilities.
  • Every proxy request is bound to a caller project identity.
  • Incoming auth, cookie, and forwarding headers are stripped by default.
  • Targets are allowlisted by exact host or safe wildcard.
  • Unexpected persistence and clone failures fail loudly.

Operational invariants

  • Project disks outlive containers.
  • Containers are stopped after short idle windows.
  • Disk free space is monitored and auto-grown before pressure.
  • New work is rejected before the host reaches critical disk usage.
  • Backups are async and observable, not part of every command path.
  • Host restart can rediscover project disks and running containers.

Rollout plan

  1. Add PROJECT_VM_BACKEND=sandbox-host adapter in the main app.
  2. Add project APIs to the Go sandbox-host without removing the existing workspace APIs.
  3. Implement clone using XFS reflink only, with timing output.
  4. Implement generic project-scoped egress capabilities.
  5. Switch dev-miguel to sandbox-host backend for Projects.
  6. Benchmark clone, cold exec, warm exec, file ops, and proxy overhead.
  7. Add disk auto-grow and backup status before any broad migration.
  8. Extract sandbox-host into a separate repository once API contracts are stable.
